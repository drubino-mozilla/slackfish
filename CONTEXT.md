# Slackfish -- Agent Context

## What This Is

Slackfish is a Firefox extension + local backend that passively captures Slack data as you browse and makes it available to AI agents via MCP and a local web UI. It intercepts Slack's own API responses in-flight -- no external tokens, no Slack admin approval, no detectable footprint.

## Architecture

```
Firefox (Slack tab)
  ├── content-ws.js ─── patches WebSocket, forwards real-time events
  ├── content.js ─────── tracks SPA navigation (pushState/popstate)
  └── background.js ──── webRequest interceptor + native messaging client
        │
        └── Native Messaging (stdin/stdout, cold start per session)
              └── slackfish_host.py ─── writes to ~/.slackfish/
                    ├── messages/{channel_id}.json
                    ├── threads/{channel_id}/{ts}.json
                    ├── channels.json
                    ├── users.json
                    ├── context.json
                    ├── self.json
                    └── events.jsonl (activity log)

Consumers:
  slackfish_mcp.py ─── MCP server (reads cache, serves to Cursor/Claude Code)
  slackfish_web.py ─── FastAPI web UI (reads cache + event log, SSE live updates)
```

### Data flow

1. **HTTP interception**: `background.js` uses `webRequest.onBeforeRequest` with `filterResponseData` to passively read Slack API response bodies. Captured methods: `conversations.history`, `conversations.view`, `conversations.replies`, `chat.postMessage`, `chat.update`, `users.info`, `users.list`, `client.boot`, and others.

2. **WebSocket interception**: `content-ws.js` patches the page's `WebSocket` constructor at `document_start`, forwarding incoming Slack real-time events (new messages, edits, user changes) to the background script.

3. **Edge API capture**: `background.js` also captures Slack's Edge API responses (search results, boot data) which use a different response format (`results` instead of `messages`).

4. **Native messaging**: Captured data is normalized and sent to `slackfish_host.py` via Firefox's native messaging protocol (4-byte length-prefixed JSON over stdin/stdout). The host process is launched cold per session -- it is NOT always-on.

5. **Cache**: The host writes JSON files to `~/.slackfish/`. Messages are deduplicated by `ts` (Slack's message timestamp, which is a unique ID). Channels and users are merged by ID.

6. **Event log**: The host appends JSONL events to `events.jsonl` for the Activity view. Events track new messages, duplicates, channel/user updates, and navigation. Rotated at 4000 lines down to 2000.

## File Map

### Extension (root of repo)

| File | Purpose |
|------|---------|
| `manifest.json` | MV2 extension manifest. Permissions: `webRequest`, `webRequestBlocking`, `nativeMessaging` on `*.slack.com` |
| `background.js` | Core logic: API interception, WebSocket event processing, native messaging client. Contains `normalizeMessage`, `normalizeUser`, `normalizeChannel`, `processApiResponse`, `processWsEvent` |
| `content-ws.js` | Content script (`document_start`): patches `WebSocket` constructor via `wrappedJSObject` to observe Slack real-time events |
| `content.js` | Content script (`document_idle`): patches `history.pushState`/`replaceState` to track Slack SPA navigation and report current channel |
| `icons/` | Extension icons (16-256px PNG) |

### Host (`host/`)

| File | Purpose |
|------|---------|
| `slackfish_host.py` | Native messaging host. Receives data from extension, writes to cache, emits activity events. Auto-registers bot users from message metadata |
| `cache.py` | `SlackfishCache` class. File-based JSON storage for channels, users, messages, threads, context. Deduplicates by `ts`/`id`, returns `(new_msgs, dup_count)` from updates |
| `protocol.py` | Native messaging wire protocol (4-byte LE length prefix + JSON) |
| `slackfish_mcp.py` | MCP server exposing cache to AI agents. Tools: `slack_list_channels`, `slack_get_messages`, `slack_get_thread`, `slack_search`, `slack_get_context`, `slack_get_stats` |
| `slackfish_web.py` | FastAPI web UI server on `localhost:7890`. REST API + SSE live updates + heartbeat auto-shutdown. Uses `watchdog` to detect cache file changes |
| `install.py` | Registers native messaging host in Windows registry (`HKCU\Software\Mozilla\NativeMessagingHosts\slackfish`) |
| `slackfish.bat` | Batch launcher for the native host (called by Firefox) |
| `slackfish.json` | Native messaging manifest (generated by `install.py`) |
| `launch-web.ps1` | PowerShell script for the desktop shortcut: launches `pythonw.exe` in background, opens `loading.html` immediately |
| `static/index.html` | Single-page web UI: channel list, message view, thread panel, search, Activity event log |
| `static/loading.html` | Startup page shown while the web server is starting (~30s). Polls `/api/stats`, redirects when ready |

## Key Design Decisions

### Cold-start native host
The native messaging host is launched by Firefox on demand (when the extension first sends a message) and dies when the extension disconnects. It is NOT a long-running daemon. This is how Firefox native messaging works -- you cannot change it.

### File-based cache, not a database
All data is stored as flat JSON files in `~/.slackfish/`. This keeps things simple, avoids dependencies, and makes the data trivially inspectable. The tradeoff is no indexing -- search is brute-force.

### Event log for Activity view
The Activity view needs to show events ordered by "indexing time" (when cached) not "message time" (when posted), and needs to show feedback for duplicate messages. This requires a separate append-only event log (`events.jsonl`) since the message cache overwrites in place.

### Bot user auto-registration
Slack bots have `B`-prefixed IDs and aren't returned by `users.list`. The host auto-registers bots into the user cache when it sees a message with a `bot_id` and `username`/`bot_profile.name`.

### Web UI heartbeat
The web server (`slackfish_web.py`) auto-exits after 60 seconds with no heartbeat from any browser tab. Tabs send `POST /api/heartbeat` every 10 seconds. This prevents orphaned server processes.

### CORS for loading page
The loading page is opened as a `file://` URL for instant feedback before the server is ready. This requires CORS (`allow_origins=["*"]`) on the FastAPI server so the page can poll `/api/stats`.

## Cache Structure (`~/.slackfish/`)

| File | Shape |
|------|-------|
| `channels.json` | `{channel_id: {id, name, is_im, is_mpim, is_private, user, topic, ...}}` |
| `users.json` | `{user_id: {id, name, real_name, display_name, email, is_bot, ...}}` |
| `messages/{channel_id}.json` | `{messages: [{ts, user, text, thread_ts, reply_count, reactions, files, subtype, bot_id, username}], channel_id, updated_at}` |
| `threads/{channel_id}/{ts}.json` | Same shape as messages, scoped to a thread |
| `context.json` | `{channel_id, url, timestamp}` -- what channel the user is currently viewing |
| `self.json` | Current user's profile |
| `events.jsonl` | Append-only JSONL. Types: `new`, `dup`, `channels`, `users`, `context`. Each has `t` (indexing time) |
| `host.log` | Debug log from the native host |

## Web UI API (`localhost:7890`)

| Endpoint | Returns |
|----------|---------|
| `GET /api/stats` | Cache statistics |
| `GET /api/channels` | All channels sorted by most recent message |
| `GET /api/channels/{id}/messages` | Messages for a channel |
| `GET /api/threads/{channel_id}/{thread_ts}` | Thread messages |
| `GET /api/search?q=term` | Full-text search across cached messages |
| `GET /api/context` | Current Slack navigation context |
| `GET /api/users` | All cached users |
| `GET /api/activity?limit=300` | Recent events from the event log (newest first) |
| `GET /events` | SSE stream: `channel_update`, `channels_update`, `users_update`, `context_update`, `activity_update` |
| `POST /api/heartbeat` | Keep the server alive |

## Extension Message Normalization

### `normalizeMessage(msg)`
```
ts, user (falls back to bot_id), text, thread_ts, reply_count,
reactions [{name, count}], files [{name, mimetype, url}],
subtype, bot_id, username (falls back to bot_profile.name)
```

### `normalizeUser(u)`
```
id, name, real_name, display_name, email, title, is_bot
```

### `normalizeChannel(ch)`
```
id, name, is_im, is_mpim, is_private, user, topic, purpose, num_members, last_read
```

## Common Tasks

### Restart the web server
Kill the process on port 7890 and relaunch. The server takes ~30 seconds to start due to Python import overhead.

### Reload the extension
Go to `about:debugging#/runtime/this-firefox`, click "Reload" on Slackfish. Required after any change to `background.js`, `content-ws.js`, `content.js`, or `manifest.json`.

### Native host changes
Changes to `slackfish_host.py`, `cache.py`, or `protocol.py` take effect automatically on the next native host cold start (next time the extension sends data). No restart needed.

### Web server changes
Changes to `slackfish_web.py` or `static/index.html` require a server restart. `index.html` is read on each request, so only Python changes technically need a restart, but a browser refresh is needed for HTML/JS changes.

### Check the host log
`~/.slackfish/host.log` has debug output from the native messaging host.

### Check the event log
`~/.slackfish/events.jsonl` has the activity event stream. `Get-Content ~/.slackfish/events.jsonl -Tail 20` to see recent events.

## Dependencies

- Python 3.12+ with `fastapi`, `uvicorn`, `watchdog` (see `host/requirements.txt`)
- Firefox with the extension loaded as a temporary add-on
- Windows (native messaging registration uses `winreg`; `launch-web.ps1` uses PowerShell)
